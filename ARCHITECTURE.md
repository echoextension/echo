## 🔬 值得一看的设计

### 🖱️ 鼠标手势的跨标签状态衔接

**问题**：用户按住右键滚轮切换标签后，焦点跳到了新标签。但新标签的 content script 不知道"右键还按着"，所以用户继续滚轮时手势失效。

**方案**：

```
Tab A (content.js)           background.js            Tab B (content.js)
    │                            │                         │
    ├─ mousedown(右键)           │                         │
    ├─ mouseGestureStart ──────►│                         │
    │                            ├─ isRightMouseDown=true  │
    │                            │                         │
    ├─ wheel(滚轮) ──────────►  │                         │
    │                            ├─ switchTab ──────────►  │
    │                            │  tabs.sendMessage        │
    │                            ├─ syncMouseGestureState─►│
    │                            │  {isRightMouseDown:true} │
    │                            │                         ├─ 本地状态同步 ✓
    │                            │                         ├─ 继续响应滚轮 ✓
```

background 维护全局 `isRightMouseDown` 标志，每次切换标签时通过 `syncMouseGestureState` 消息将状态推送到新标签。新标签同时设置 `preventContextMenu = true`，确保松开右键时不弹出菜单。

📍 代码位置：[content.js](content.js) L56-L130、[background.js](background.js) L80-L88

### 📋 新标签位置：Promise 队列解决并发竞态

**问题**：用户快速连续点击多个链接（比如在搜索结果页 Ctrl+Click 连开 5 个），`chrome.tabs.onCreated` 事件近乎同时触发。如果不做处理，多个 `tabs.move()` 会互相踩踏，导致标签乱序。

**方案**：用 `tabCreationQueue`（per-window 的 Promise 链）串行化处理：

```javascript
// 每个窗口独立的 Promise 队列
const tabCreationQueue = new Map(); // Map<windowId, Promise>

chrome.tabs.onCreated.addListener((tab) => {
  const windowId = tab.windowId;

  // 关键：在 onCreated 的同步时刻立即捕获 baseTabId
  // 此时 onActivated 尚未触发，state.baseTabId 还是父标签
  const snapshotBaseTabId = state.baseTabId;

  // 将处理任务加入 Promise 链，确保顺序执行
  const currentQueue = getWindowQueue(windowId);
  const newQueue = currentQueue.then(() => handleNewTabCreated(tab, snapshotBaseTabId));
  tabCreationQueue.set(windowId, newQueue);
});
```

两个关键点：

- **同步快照**：在 `onCreated` 触发的同步时刻立即抓取 `baseTabId`，避免后续 `onActivated` 覆盖掉真正的"父标签"
- **Promise 链串行**：同一窗口的所有 `tabs.move()` 通过 `.then()` 链串行执行，彻底消除竞态

此外还处理了 `onMoved` 事件——当用户拖拽标签时同步更新 `baseTabIndex`，并区分"向左拖"和"向右拖"对索引的不同影响。

📍 代码位置：[background.js](background.js) L248-L510

### 🖼️ 壁纸系统的三级缓存

**问题**：NTP 每次打开都要加载高清壁纸，如果每次都走网络请求，白屏时间明显。

**方案**：Memory → IndexedDB → Network 三级回退：

```
打开新标签页
    │
    ▼
┌─────────────────┐
│ 1. 内存缓存      │ ← 当前会话已加载的 Blob URL
│   (变量引用)      │    命中 → 直接渲染，0 延迟
└────────┬────────┘
         │ 未命中
         ▼
┌─────────────────┐
│ 2. IndexedDB     │ ← echo_wallpaper_cache 数据库
│   (Blob 离线)    │    keyPath: url, 7天TTL自动清理
│                   │    命中 → createObjectURL → 渲染
└────────┬────────┘
         │ 未命中
         ▼
┌─────────────────┐
│ 3. 网络请求      │ ← Bing 壁纸 API
│   (fetch+Blob)   │    成功 → 同时写入内存 + IndexedDB
└─────────────────┘
```

IndexedDB 存储原始 Blob（非 URL 字符串），通过 `cleanOldWallpaperCache()` 每次启动时清理超过 7 天的条目。这样即使完全离线，只要 7 天内打开过 NTP，壁纸仍然秒加载。

### 🛡️ Shadow DOM 隔离与缩放补偿

**问题**：悬浮搜索框和 AI 关联搜索需要注入到任意网页，但不能被宿主页面的 CSS 影响（反之亦然）。更棘手的是，当用户 Ctrl+滚轮缩放页面时，注入的 UI 也会跟着变大/变小。

**方案**：

**隔离**：使用 Closed Shadow DOM：

```javascript
const host = document.createElement('div');
const shadow = host.attachShadow({ mode: 'closed' });
// 外部 JS 无法通过 host.shadowRoot 访问内部结构
```

**缩放补偿**：定期轮询页面缩放级别，计算反向 scale：

```javascript
function applyZoomCompensation(zoomLevel) {
  const inverseScale = 1 / zoomLevel;

  // 反向缩放：页面放大200%时，UI缩小到50%，视觉上大小不变
  host.style.transform = `translateX(-50%) scale(${inverseScale})`;

  // 位置补偿：bottom 距离也要反向计算
  // 物理像素距离 = bottom(CSS px) × zoomLevel
  // 要保持物理距离恒定：bottom = 目标距离 / zoomLevel
  host.style.setProperty('--echo-bottom', `${BOTTOM_OFFSET_PX * inverseScale}px`);
}
```

注意不仅要补偿 `scale`，还要补偿 `bottom` 定位——否则页面放大后搜索框会"飘"到屏幕外。这里用 CSS 变量 `--echo-bottom` 动态修正。

轮询间隔 500ms，通过 `chrome.tabs.getZoom()` （经 background 中转）获取精确值。

📍 代码位置：[search-box/search-box.js](search-box/search-box.js) L550-L645

### 🤖 AI 响应的三策略容错解析器

**问题**：免费 AI API（Pollinations.ai / Ollama）的输出格式极不稳定——可能返回标准 JSON 数组、嵌套对象、Markdown 代码块包裹的 JSON、带编号的纯文本列表、甚至混合格式。Prompt 要求"纯文本列表"，但实际返回经常不遵守。

**方案**：三层瀑布式解析，逐层降级：

```
AI 原始响应
    │
    ▼
┌─────────────────────────────────────┐
│ 策略 A：JSON 结构化解析              │
│  - 去除 ```json ``` 包裹             │
│  - 正则提取最外层 {} 或 []           │
│  - 数组 → flatMap 提取字符串         │
│  - 对象 → 找 queries/keywords 等     │
│    候选 key，或递归提取值             │
│  - 清洗推理链字段(reasoning等)       │
│  - 处理 {"content":"Line1\nLine2"}  │
│    这种单字段多行值的情况             │
│  - Values 无效时尝试用 Keys 作关键词  │
└──────────┬──────────────────────────┘
           │ 失败
           ▼
┌─────────────────────────────────────┐
│ 策略 B：正则引号提取                  │
│  - 匹配所有 "..." 引号内容           │
│  - 过滤掉 JSON key（含":"）          │
│  - 过滤 "assistant"/"user" 等角色标记│
└──────────┬──────────────────────────┘
           │ 失败
           ▼
┌─────────────────────────────────────┐
│ 策略 C：纯文本行分割                  │
│  - 去除代码块包裹                     │
│  - 按换行split                       │
│  - 正则去除行首序号(1. 2) 3-)和bullet│
│  - 去除两端引号                       │
└─────────────────────────────────────┘
           │
           ▼
┌─────────────────────────────────────┐
│ 统一后处理                            │
│  - 去重（Set）                        │
│  - CJK 检测：中文 4-35 字，英文需含   │
│    空格且 10-100 字符                  │
│  - 过滤纯标点/数字                    │
│  - 黑名单词过滤(null/keywords等)      │
│  - 防标题重复（与 document.title 比对）│
│  → 最终保留 ≥3 个有效关键词            │
└─────────────────────────────────────┘
```

策略 A 内部还有一个特别的容错：当 JSON 对象的 Values 全部为空字符串（如 `{"关键词1":"", "关键词2":""}`）时，会反转思路用 Keys 作为关键词提取——因为某些模型会把关键词写在 key 而非 value 里。

📍 代码位置：[related-search/related-search.js](related-search/related-search.js) L168-L405

### 🔎 精细缩放的双阈值系统

**设计观察**：浏览器原生缩放步进在高倍率下太粗（110% → 125% → 150% → 175% → 200%），在接近 100% 的常用区间又太大（一步就跨了 10-25%）。

**方案**：以 175% 为分界线的双阈值：

| 缩放范围   | 步进  | 理由                         |
| ------ | --- | -------------------------- |
| ≤ 175% | 5%  | 常用区间，需要精细调节（90→95→100→105） |
| > 175% | 25% | 高倍率下精细调节无意义，大步进更高效         |

175% 这个数字不是随便选的——它是浏览器原生步进列表中"精细需求"与"快速跳转需求"的自然分界点。

📍 代码位置：[content.js](content.js) 缩放处理段

### 📊 内容信噪比检测

**问题**：AI 关联搜索应该只在"有实质内容的文章页"触发，不应该在首页、搜索结果页、导航页上浪费 API 调用。但怎么判断一个页面是不是"文章页"？

**方案**：除了 URL 黑名单（.gov/.mil/.edu/搜索引擎等），还有一个**内容信噪比检测**：

```javascript
const lines = content.split('\n');
const validLines = lines.filter(l => l.trim().length > 0);
const avgLineLen = validLines.reduce((acc, l) => acc + l.length, 0) / (validLines.length || 1);
const hasLongParagraph = lines.some(l => l.length > 80);

// 聚合页特征：总字数不少，但全是短标题，没有长段落
if (!hasLongParagraph && avgLineLen < 30) {
  // 跳过分析
}
```

逻辑：文章页一定有"长段落"（>80字的行），如果一个页面提取出来全是短行（平均 <30 字）且没有任何长段落，大概率是标题聚合页（如 B 站首页、新闻列表页），跳过。

这一检测在 URL 过滤之后、API 调用之前执行，**不消耗 API 配额**。

📍 代码位置：[related-search/related-search.js](related-search/related-search.js) L190-L202

## 💡 关于这个项目

### 👤 关于作者

ECHO 是一个非技术背景的 PM 通过 AI 辅助编程（Vibe Coding）独立完成的个人饭制作品，**非** Microsoft 官方产品，**非** Edge 团队出品。作者没有写过一行生产级代码，但对中国互联网特别是浏览器产品有超过十年的行业经验——ECHO 中每一个功能的取舍，都不是凭空想象。

上面这些技术方案并非一次成型，而是在反复使用和调试中逐步演化出来的——有些是为了修复"快速开 5 个标签后顺序乱了"这样的实际 bug，有些是为了让注入 UI 在任何缩放级别下都不变形，有些纯粹是对"体验应该怎样"的执念。

如果你有兴趣贡献代码或提出改进，希望这份文档能帮你快速理解两件事：代码**是怎么写的**（How），以及**为什么要这样写**（Why）。

说到 Why——为什么一个 PM 要自己动手做一个浏览器扩展？

### 🌏 心系天下

Edge 在中国是市场占有率第一的浏览器，这体现了某种理性的选择。

身处这片特殊的数字土壤，用户往往需要在两个极端之间寻找平衡：**一端是体验纯粹但服务链路时常受阻的全球化标杆，另一端是功能贴地但有时在注意力争夺上过于“热情”的本地化产品。** Edge 恰好站在了那个难得的交汇点上——它拥有 Chromium 的先进内核与生态兼容，保持了难得的体面与克制，同时还有微软持续投入的 AI 能力加持。

但这种全球化的体面也带来了一个缺憾：它很难完全俯身去适配每一个区域的肌肉记忆。而中国互联网有一段独特的浏览器进化史。从傲游（Maxthon）开创多标签浏览开始，鼠标手势、超级拖拽、快捷存图、标签精细管理……这些功能在国产浏览器的竞争中被反复打磨，最终沉淀为数以亿计用户的**肌肉记忆**。这不是极客的小众偏好，而是一代人被十几年使用习惯训练出的本能反应——右键划一个 L 就是关闭标签，拖拽选中文字就是搜索，新标签页应该打开在当前标签旁边而不是最右端。

我们不能要求一个国际化产品为每个区域的历史习惯逐一适配，也不应该要求用户放弃自己的使用直觉去"重新学习"。最好的方式是**双向奔赴**——让浏览器更懂用户，也让用户更容易拥抱更好的浏览器。

ECHO 试图解决的正是这个问题。

市面上并不缺单一功能的扩展——有做鼠标手势的，有做新标签页美化的，有做快捷存图的，有支持一些快捷键的，甚至极个别的可以调整标签打开逻辑。但把大约十个核心交互差异**完整地组装到一起**，提出"让 Edge 更懂中国用户"这个明确主张，并且为每一项功能都提供独立开关让用户精细调控的——ECHO 很可能是第一个这样做的。

而且这绝不仅仅是为极客和硬核用户准备的。试想一个普通用户——他不知道什么是 MV3 或 Service Worker，但他知道"以前的浏览器右键拖一下就能关标签，现在这个不行"。如果 ECHO 能消除这一个个微小的摩擦点，让他无缝地保留自己的使用习惯，同时享受 Edge 的安全、性能和 AI 能力——这不是一个极客工具的价值，而是帮助一个主流浏览器真正赢得中国用户的心。

这或许是一种小小的担当：不是抱怨"为什么不给我们做"，而是自己动手，为理解者建桥，为探索者点灯。

## 🧬 碳硅融合：当 AI 写代码，人注入品味

> *"In the age of AI, taste is the last moat."*

这是一个构建在“不合时宜”之上的产品。

如果要用一句自嘲的梗来形容，它充满了“**瓦坎达振金长矛**”式的荒诞美学：拥有了当今最先进的 AI 生产力（Vibranium），最终却用它打磨了一把最原始的冷兵器（鼠标手势）。

在功能维度，它显得**传统而克制**。当行业都在追逐大模型的通用智能时，ECHO 选择回头去解决那些最朴素的遗留问题——因为我们认为，**浏览器的基础交互（如手势）不应该因为 AI 的到来而被遗忘**。降低基础操作的摩擦力，本身就是为用户接纳新技术铺路。

在生产维度，它又是**激进的试验品**。这是一个几乎完全由 AI 编写的代码库。作者不再是逐行编码的工匠，而是逻辑的编排者。这代表了一种新的生产关系：**AI 负责穷尽实现的可能，而人类负责收敛选择的边界。**

**由此，构成了“碳硅融合”的双重现实：**

**第一重是外在的：** 利用最前沿的 AI 生产力，去修复最基础的用户体验——**即便是不使用大模型的用户，也能间接享受到技术变革带来的效率红利。**

**第二重是内在的：** 当 AI 接管了代码实现的繁琐，人类的核心价值就回归到了**判断力与审美**。

AI 可以写出运行无误的代码，但它不会主动在意那些“需求文档之外”的细腻触感。在 ECHO 的开发过程中，AI 负责“如何做”，而人负责决定“做什么”。以下是一些例子——它们在功能上或许不是必需的，但正是这些细节界定了“产品”与“工具”的区别。

### 🎨 壁纸主色提取与全局色彩自适应

NTP 有**两条独立的颜色采样管线**，各自负责不同的 UI 区域：

**管线 1：信息卡片区域**（`extractAndApplyWallpaperTheme`）——Canvas 采样壁纸**左上角**（卡片所在位置，约 300×150 像素区域），通过颜色桶量化（每 32 级一个桶）找出主色调，然后：

- 主色经过饱和度增强（`boost = 200 / maxChannel`）后，生成鲜艳版本作为卡片背景渐变的左侧点缀色
- 根据采样区亮度动态切换手柄、边框、光晕的配色方案（亮壁纸用主题色深版 + 白边框，暗壁纸用主题色亮版 + 深边框）
- 卡片背景本身是 `backdrop-filter: blur(12px)` 毛玻璃 + 左侧主题色渐变叠加

### 💫 搜索框的聚焦脉冲：三层椭圆光环

当用户按 Ctrl+B 唤出悬浮搜索框时，会播放一个 0.4 秒的椭圆光环脉冲扩散动画——三层光环（蓝 `#38bdf8`、紫 `#c084fc`、粉 `#f472b6`）依次扩散然后消失。

这个动画有几个细节值得注意：

- 光环形状精确匹配搜索框的尺寸和圆角，并且**根据热搜面板是否展示动态调整宽度**（710px vs 420px）
- 光环必须放在 Shadow DOM **外部**（直接挂在 `document.body` 上），因为光环要覆盖全屏，不能被 Shadow DOM 的 `overflow` 裁剪
- 当页面有缩放时，光环也要做反向缩放补偿，否则光环和搜索框的大小/位置会错位

这个动画总共持续不到 1 秒，可能 90% 的用户不会注意到它的存在。但它给了"搜索框弹出"这个瞬间一种"呼应感"，从"弹出一个框"变成了"这个框在向你打招呼"。
